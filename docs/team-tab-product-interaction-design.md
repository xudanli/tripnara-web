# 行程详情页 - 团队 Tab 产品与交互设计方案

**目标受众**：产品经理、交互设计师、前端开发  
**关联模块**：行程详情页 `[id].tsx` - TeamTabContent、TeamManagementPanel、CreateTeamDialog  
**设计原则**：结论优先 | 任务清晰 | 降低认知负担 | 与「协商结果-视觉方案」保持一致

**产品定位**：团队 Tab 为团队能力的**唯一主入口**，与规划 Tab 智能优化（三守护者协商）职责分离。详见 `docs/team-tab-product-manager-decision.md`。

**后端接口**：邀请等能力所需接口详见 `docs/api/team-tab-backend-requirements.md`。

---

## 一、现状分析

### 1.1 当前页面结构

| 状态 | 内容 | 交互 |
|------|------|------|
| **未关联团队** | 空状态卡片：图标 +「创建行程团队」+ 文案 + 主按钮 | 点击「创建团队」打开 CreateTeamDialog |
| **已关联团队** | ① 团队协商 Card ② 团队成员 Card | 发起协商、添加/移除成员、折叠区块 |

### 1.2 当前痛点

| 问题 | 表现 | 影响 |
|------|------|------|
| 信息层级混乱 | 协商结论、成员评估、冲突详情、约束、权重混在同一流 | 用户难以聚焦「下一步该做什么」 |
| 协商结果呈现粗糙 | 仅 Badge + 数字，缺少结构化任务 | 与规划 Tab 的 NegotiationResultCard 风格不一致 |
| 创建→管理断层 | 创建团队后直接进入管理面板，缺少「邀请」引导 | 新团队无成员时体验断层 |
| 团队约束/权重默认折叠 | 重要约束藏在折叠区，易被忽视 | 可能影响行程安全与共识 |
| 无「邀请」闭环 | 仅有添加成员，无分享链接/二维码 | 实际协作依赖外部沟通 |

## 二、用户旅程与任务分解

### 2.1 典型用户旅程

```
进入行程详情 → 点击「团队」Tab → 
  ├─ 无团队：创建团队 → 添加成员 → (可选) 发起协商
  └─ 有团队：查看状态 → 添加/移除成员 → 发起协商 → 查看结果 → 执行建议
```

### 2.2 核心任务优先级

| 优先级 | 任务 | 当前支持 | 建议 |
|--------|------|----------|------|
| P0 | 创建/关联团队 | ✓ | 优化空状态与引导 |
| P0 | 管理成员（增删） | ✓ | 保持，优化成员卡片 |
| P0 | 发起团队协商 | ✓ | 明确前置条件与反馈 |
| P1 | 理解协商结果并行动 | 部分 | 复用 NegotiationResultCard 风格 |
| P1 | 查看团队约束 | ✓ 折叠 | 高亮「最弱链」来源 |
| P2 | 邀请他人加入 | ✗ | 增加分享/邀请能力 |
| P2 | 编辑成员偏好 | ✓ 部分 | 完善编辑流 |

## 三、信息架构（IA）重构

### 3.1 推荐页面分区（自上而下）

| 分区 | 内容 | 展示规则 |
|------|------|----------|
| **L1 状态区** | 团队名称、成员数、协商状态 | 始终展示 |
| **L2 行动区** | 发起协商 / 邀请成员 | 有团队时展示 |
| **L3 协商结果** | 共识度、结论、可执行建议 | 有协商结果时展示 |
| **L4 成员与约束** | 成员列表、团队约束、权重 | 可折叠，约束满足时默认展开 |

### 3.2 空状态（未关联团队）

- 保持当前「创建行程团队」卡片
- 增加副标题或说明：「创建后可邀请同行者，共同设定偏好并协商行程」
- 可选：增加「已有团队？关联行程」入口（若产品支持团队复用）

## 四、交互设计方案

### 4.1 发起协商流程

| 步骤 | 交互 | 反馈 |
|------|------|------|
| 1 | 用户点击「发起协商」 | 校验：成员≥1、有行程计划 |
| 2 | 校验失败 | Toast + 引导（如「请先添加至少 1 名成员」） |
| 3 | 校验通过 | Loading：「协商中...」 |
| 4 | 协商完成 | 展开 L3 协商结果区，可选 Toast「协商完成」 |

### 4.2 协商结果展示（对接协商结果视觉方案）

参考 `.claude/agents/协商结果-视觉方案.md`，统一信息层级与组件风格：

| 元素 | 规范 |
|------|------|
| 共识度 + 投票 | L1，14px，核心决策状态 |
| 评估维度卡片 | L2，12px 标签 + 14px 数字 |
| 建议任务行 | L3，SuggestionTaskRow 组件，48px 高 |
| 冲突详情 | 独立区块，与「建议你先做」分离 |

**建议**：将 TeamTab 内协商结果区抽离为与 NegotiationResultCard 同源的子组件，复用 SuggestionTaskRow、维度标签映射等。

### 4.3 成员管理

| 操作 | 当前 | 建议 |
|------|------|------|
| 添加成员 | AddMemberDialog 弹窗 | 保持，增加「继续添加」快捷入口 |
| 移除成员 | 图标按钮 | 增加二次确认（非领队时） |
| 编辑成员 | 设置图标 | 完善编辑流：角色、体能、权重、约束 |
| 邀请 | 无 | 新增：分享链接 / 二维码（P2） |

### 4.4 团队约束与权重

| 区块 | 默认状态 | 说明 |
|------|----------|------|
| 团队约束（最弱链） | 默认展开 | 安全相关，应优先可见 |
| 约束来源 | 每条约束旁标注来源成员 | 帮助理解「谁限制了团队」 |
| 权重分配 | 默认折叠 | 高级信息，需时再展开 |

## 五、视觉与 Token 规范

### 5.1 与协商结果方案对齐

```
--suggestion-row-height: 48px;
--suggestion-gap: 12px;
--task-summary-max: 28 字;
--action-button: 32px 高, text-xs;
```

### 5.2 状态 Badge 映射

| 协商结论 | Badge 样式 | 说明 |
|----------|------------|------|
| APPROVE | default（通过） | 绿色系 |
| REJECT | destructive（拒绝） | 红色系 |
| APPROVE_WITH_CONDITIONS | secondary（有条件通过） | 黄色/琥珀 |
| 约束满足 | outline | 中性 |
| 存在冲突 | destructive | 需关注 |

### 5.3 成员卡片

- 保持 Avatar + 姓名 + 角色图标 + 体能 Badge + 权重
- 操作区：编辑 | 移除（领队不可移）

## 六、边界与降级

| 场景 | 处理 |
|------|------|
| 成员为 0 | 提示「添加成员后再发起协商」 |
| 无行程计划 | 提示「请先在规划 Tab 生成行程」 |
| 协商失败 | Toast + 保留上一次协商结果（若有） |
| 团队加载失败 | 显示错误态，提供重试 |
| 暗色模式 | 图标与背景使用 opacity、border 区分 |

## 七、实施建议（按阶段）

### Phase 1：信息层级与协商结果统一（P1）

1. 重构 Team Tab 页面分区，按 L1–L4 排布  
2. 将协商结果区复用/对齐 NegotiationResultCard 的 SuggestionTaskRow、维度映射  
3. 团队约束默认展开，并突出约束来源  

### Phase 2：流程与引导优化（P1）

1. 发起协商前校验（成员、计划）  
2. 新创建团队后的引导（如「添加第一位成员」）  
3. 成员移除二次确认  

### Phase 3：邀请与协作增强（P2）

1. 分享链接 / 二维码邀请 — 已加「邀请成员」入口与占位对话框，待后端就绪接入  
2. 成员编辑流程完善 — 已落地 EditMemberDialog
