# 自然语言创建行程：对话进行中用户终止能力 - 产品经理评估

## 一、需求描述

**提议**：在对话进行中（AI 正在思考/生成回复时），用户可随时终止当前轮次。

## 二、当前实现状态

| 维度 | 现状 |
|-----|------|
| **前端** | `isLoading` 期间无「终止」按钮；用户无法中止正在进行的请求 |
| **API** | `tripsApi.createFromNL` 为单次 HTTP POST，未支持 `AbortController.signal` |
| **已有终止** | 仅「自动提交倒计时」提供「取消」按钮；主对话轮次无终止能力 |

## 三、产品评估

### 3.1 支持「随时终止」的理由

| 维度 | 说明 |
|-----|------|
| **用户自主权** | 用户发现描述有误、或改变主意时，应能立即停止，不必等待数十秒 |
| **避免无效等待** | NL 创建行程单次请求可达 60–120 秒，误触或错误输入后的等待成本高 |
| **行业惯例** | ChatGPT、Claude、Gemini 等均支持 Stop / 终止生成 |
| **TripNARA 原则** | 「Friction is intentional」—— 在需要确认时有摩擦；但不应在「想停止」时制造无谓摩擦 |

### 3.2 需要关注的风险与边界

| 风险/边界 | 说明 | 建议 |
|----------|------|------|
| **已提交消息是否保留** | 终止后，用户刚发送的那条消息已在列表中。若完全回滚，会产生「发了又没发」的认知混乱 | **保留用户消息**，仅不追加 AI 回复；或追加一条「已取消」的占位提示 |
| **后端已部分执行** | 若后端为流式/分步执行，终止时可能已有部分结果写入 DB | 前端终止只影响本端，不要求后端回滚；后续请求不受影响 |
| **误触终止** | 用户可能误点「终止」，导致需要重新发送 | 终止前可无需二次确认（与竞品一致）；误触后重发即可 |
| **会话一致性** | 终止后 sessionId 是否仍有效、下一轮能否正常继续 | 终止仅取消本次请求，session 保持；下一轮正常发送 |

### 3.3 逻辑是否合理：结论

**结论：支持「用户随时可终止」逻辑合理，建议实现。**

理由：
1. **符合用户预期**：长耗时任务的取消是通用心智模型。
2. **风险可控**：保留用户消息、session 不变、不要求后端回滚，实现成本低。
3. **与现有能力一致**：自动提交倒计时的「取消」已存在，主对话轮次的终止是其自然延伸。

### 3.4 实施要点（供开发参考）

1. **前端**：`sendMessage` 使用 `AbortController`，在「终止」按钮点击时调用 `abort()`。
2. **UI**：`isLoading` 时在输入区附近展示「终止」按钮（如 ⬛ 或 Stop 图标）。
3. **状态**：终止后 `setIsLoading(false)`，可选追加一条 assistant 消息如「已取消」或不再追加（保持当前对话尾部为用户消息）。
4. **API**：`createFromNL` 支持传入 `signal`，并传递给 axios/fetch。

## 四、附录：与自动提交「取消」的对比

| 场景 | 行为 | 说明 |
|------|------|------|
| 自动提交倒计时 | 用户点「取消」→ 停止倒计时、不发送 | 已实现 |
| 主对话 loading | 用户点「终止」→ 取消当前 HTTP 请求、不追加 AI 回复 | 待实现 |
