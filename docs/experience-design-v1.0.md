## 引言

**体验设计的核心使命：** 让用户通过视觉、交互和信息流的精心设计，进入"判断→执行→信任"的决策节奏。

这份文档回答的问题是：

**如何设计一个界面系统，既能展现复杂的决策信息，又能让用户感受到清晰的逻辑、自主的权力和可信的过程？**

TripNARA的体验设计不是追求美观，而是追求"在现实约束下依然可信"的视觉表达。每个设计决策都服务于一个目标：帮助用户判断，而不是代替用户判断。

## 第一章：全局视觉系统设计

### 1.1 设计哲学

TripNARA的视觉系统建立在三个哲学基础上：

#### 1.1.1 结构感优于装饰感

在TripNARA中，所有的视觉设计都是为了帮助用户理解**信息的层级和逻辑关系**，而非追求视觉的华丽。

```
错误的做法：
├─ 大量图片和视频，让界面看起来"丰富"
├─ 复杂的动画，让界面看起来"精致"
└─ 精美的图标和插画，让界面看起来"友好"

正确的做法：
├─ 清晰的信息架构，让用户快速理解
├─ 有效的对比，帮助用户做出判断
├─ 明确的信息来源标注，让用户相信数据
└─ 预留充足的"思考空间"，让用户自己判断
```

#### 1.1.2 情感感与理性感的平衡

用户看到TripNARA的界面，既要有"这个系统理解我"的感受，又要有"这个系统是可靠的"的感受。

```
情感层面：
├─ 温暖的色调和间距，表达同理心
├─ 人性化的语言和措辞，表达尊重
└─ 进度反馈和鼓励，表达支持

理性层面：
├─ 严谨的数据呈现，表达专业性
├─ 清晰的逻辑链条，表达可靠性
└─ 透明的决策依据，表达可信性
```

#### 1.1.3 渐进式信息披露

用户不应该被信息淹没。系统采用"分层呈现"策略：

```
第一层（快速感知）：核心答案
  "这条路线安全吗？" → "中等风险"

第二层（理解）：为什么
  "中等风险，因为..."（3-5个关键因素）

第三层（深度）：如何判断
  完整的数据、假设、限制条件

用户可以在任何层停留，也可以深入下去。
```

### 1.2 色彩系统

#### 1.2.1 色彩的功能性

TripNARA的色彩不是装饰，而是信息的编码。

```
【色彩的四个角色】

1. 安全色（绿色系）
   含义：安全、建议、正向
   使用场景：
   ├─ 风险评分：低风险 → 绿色
   ├─ 决策建议：推荐 → 绿色
   ├─ 按钮：确认 → 绿色
   RGB: #4CAF50 (主色)
   Range: #45a049 - #66BB6A

2. 警告色（黄色系）
   含义：注意、需要考虑、中风险
   使用场景：
   ├─ 风险评分：中等风险 → 黄色
   ├─ 消息：需要注意 → 黄色
   ├─ 不确定性：高不确定性 → 黄色
   RGB: #FFC107 (主色)
   Range: #FFB300 - #FFE082

3. 风险色（红色系）
   含义：危险、不推荐、高风险
   使用场景：
   ├─ 风险评分：高风险 → 红色
   ├─ 禁止：不可行 → 红色
   ├─ 错误：系统错误 → 红色
   RGB: #F44336 (主色)
   Range: #EF5350 - #EF9A9A

4. 中立色（蓝/灰色系）
   含义：信息、中立、数据
   使用场景：
   ├─ 信息按钮：了解更多 → 蓝色
   ├─ 数据呈现：客观信息 → 灰色
   ├─ 次要操作：灰色
   RGB: #2196F3 (蓝色), #9E9E9E (灰色)
```

#### 1.2.2 颜色应用规范

```
【风险评分与颜色的对应】

低风险（0-30）：绿色
├─ 主色：#4CAF50
├─ 浅色背景：#E8F5E9
└─ 含义：安全，建议进行

中低风险（31-45）：浅绿-黄绿
├─ 主色：#9CCC65
├─ 浅色背景：#F1F8E9
└─ 含义：基本安全，可考虑

中等风险（46-60）：黄色
├─ 主色：#FFC107
├─ 浅色背景：#FFF9C4
└─ 含义：需要注意，做好准备

中高风险（61-75）：橙黄
├─ 主色：#FF9800
├─ 浅色背景：#FFF3E0
└─ 含义：风险可控但明显，需谨慎

高风险（76-90）：橙色
├─ 主色：#FF6F00
├─ 浅色背景：#FFE0B2
└─ 含义：风险高，需充分准备

极高风险（91-100）：红色
├─ 主色：#F44336
├─ 浅色背景：#FFEBEE
└─ 含义：严重风险，不推荐
```

### 1.3 排版系统

#### 1.3.1 字体策略

TripNARA使用清晰、易读的字体，优先考虑可访问性。

```
【字体选择】

标题（H1）：
├─ 字体：PingFang SC / Helvetica Neue
├─ 字号：32px (桌面) / 24px (移动)
├─ 行高：1.2
├─ 字重：600 (Semibold)
├─ 用途：页面主标题，强调决策主题
└─ 示例：你的旅行决策

次标题（H2）：
├─ 字体：PingFang SC / Helvetica Neue
├─ 字号：24px (桌面) / 18px (移动)
├─ 行高：1.3
├─ 字重：600
└─ 用途：章节标题，分区域标题

小标题（H3）：
├─ 字体：PingFang SC / Helvetica Neue
├─ 字号：18px (桌面) / 16px (移动)
├─ 行高：1.4
├─ 字重：500 (Medium)
└─ 用途：数据标签，小区域标题

正文（Body）：
├─ 字体：PingFang SC / Helvetica Neue
├─ 字号：16px (桌面) / 14px (移动)
├─ 行高：1.6
├─ 字重：400 (Regular)
├─ 颜色：#333333
└─ 用途：主要内容、解释文本

辅助文本（Caption）：
├─ 字体：PingFang SC / Helvetica Neue
├─ 字号：12px
├─ 行高：1.4
├─ 字重：400
├─ 颜色：#999999
└─ 用途：数据来源、时间戳、注释
```

#### 1.3.2 排版规范

```
【间距系统】

基础单位：4px

常用间距：
├─ 极小：4px (标签内距)
├─ 小：8px (相关元素间)
├─ 中：12px (相邻卡片间)
├─ 大：16px (区域间)
├─ 超大：24px (主要分区间)
└─ 页面边距：16px (移动) / 24px (桌面)

【行高规则】

文本长度    建议行高    理由
────────────────────────────
<30字       1.2-1.3     短文本
30-50字     1.4-1.5     中等文本
>50字       1.5-1.6     长文本
（确保可读性）

【对齐规则】

数据呈现：右对齐
  - 便于数值对比

文本内容：左对齐
  - 便于阅读

标签：根据内容上下文
  - 关键指标标签：上方或左侧对齐
```

### 1.4 组件库系统

TripNARA的设计系统包含以下核心组件：

```
【基础组件】

├─ 显示行程进度

└─ 与实时数据同步

对比表 (Comparison Table)：
├─ 多选项对比
├─ 强调关键差异
└─ 支持自定义对比维度

地形剖面图 (Elevation Profile)：
├─ 显示路线高度变化
├─ 标注关键点
└─ 与用户体力对比
```

## 第二章：路线感知设计

### 2.1 地图交互设计

路线感知设计的核心是让用户能够直观理解"这条路线是什么样的"。

#### 2.1.1 基础地图层

```
【地图层级结构】

底层：卫星影像或地形图
  ├─ 让用户看到真实的地形和环境
  └─ 增强真实感和信任度

中层：路线展示
  ├─ 主路线：突出的粗线
  ├─ 备选路线：虚线
  └─ 已完成路线：淡色处理

顶层：关键信息点
  ├─ 起点/终点：明确标注
  ├─ 关键景点：图标+名称
  ├─ 风险点：警告图标
  └─ 休息点：设施图标

交互层：
  ├─ 点击景点：查看详情
  ├─ 长按路线：显示该段详情
  └─ 拖动视角：自由探索
```

#### 2.1.2 高度感知设计

用户需要感知路线的高度变化。

```
【高度剖面图设计】

视觉表现：
├─ X轴：距离（公里）
├─ Y轴：高度（米）
├─ 线条宽度：表示难度
│  ├─ 细线：平缓
│  ├─ 中线：中等
│  └─ 粗线：陡峭
├─ 线条颜色：
│  ├─ 绿色：上升
│  ├─ 红色：下降
│  └─ 灰色：平缓
└─ 背景阴影：显示崎岖程度

关键标注：
├─ 最高点：用★标注
├─ 休息点：用◆标注
├─ 危险段：用⚠标注
└─ 景观点：用◆标注

交互特性：
├─ 悬停显示：该点详细信息
├─ 点击显示：该段路线特性
├─ 滑动浏览：时间方向上的变化
└─ 对比功能：与用户体力的对比

## 第三章：决策界面设计

### 3.1 决策流程的视觉表现

#### 3.1.1 三层决策漏斗

TripNARA将用户的决策过程分为三层，每层有不同的视觉处理：

```
【第一层：浏览（Browse）】

用户还在探索，还没有形成倾向

视觉特点：
├─ 卡片展示：并排显示多个选项
├─ 信息量：中等（不过度）
├─ 交互：可点击查看详情
├─ 色彩：中立色，没有"推荐"标记
└─ 目标：帮用户快速扫描和初步判断

┌──────────────────────────────────────────┐
│  选项A      选项B      选项C     选项D   │
├──────────┬──────────┬──────────┬────────┤
│ 景观好   │ 难度中   │ 时间少   │ 轻松   │
│ 难度高   │ 景观好   │ 景观好   │ 景观   │
│ 风险高   │ 风险中   │ 风险中   │ 一般   │
│          │          │          │        │
│ 点击看   │ 点击看   │ 点击看   │ 点击看 │
│ 详情     │ 详情     │ 详情     │ 详情   │
└──────────┴──────────┴──────────┴────────┘

【第二层：理解（Understand）】

用户点击了某个选项，开始深入理解

视觉特点：
├─ 页面展开：显示完整详情
├─ 信息量：大量（全面介绍）
├─ 对比：与用户状况的对比
├─ 推荐标记：若匹配，显示"推荐"
├─ 风险提示：突出需要注意的地方
└─ 目标：让用户充分理解这个选项

┌─────────────────────────────────────────┐
│ ◀ 返回选择                              │
├─────────────────────────────────────────┤
│ 路线名称：天子山核心路线                │
│                                         │
│ 推荐度：★★★★★ (vs你的状态)           │
│                                         │
│ 核心特征：                              │
│ ├─ 景观评分：5/5 ⭐⭐⭐⭐⭐            │
│ ├─ 难度评级：困难(4/5)                 │
│ ├─ 总距离：15km                         │
│ └─ 爬升/下降：1200m / 1200m            │
│                                         │
│ 你适合吗？                              │
│ ✓ 时间充足（5天）                      │
│ ⚠ 体力紧张（你6/10, 需7/10）          │
│ ✓ 景观完全符合你的兴趣                 │
│                                         │
│ 我的建议：                              │
│ 如果你最近有锻炼，可以尝试。            │
│ 但要做好以下准备...                    │
│                                         │
│ 【查看完整信息】 【看备选方案】 【选择这个】

        "这个系统会帮我吗？" 快速展示价值

        "选这个对吗？"      "这是个好选择，

兴奋→信任：

  ✓ 进度反馈（"你已经完成60%"）
  ✓ 积极的语言
  ✓ 简化关键信息，暂时隐藏细节

决策→支持：
  ✓ 让用户知道选择很正常
  ✓ 提供后续支持
  ✓ 表达对用户选择的尊重

### 7.2 决策支持文案规范

```
【系统与用户的对话语言】

原则1：不命令，而是支持
错误："你应该选择路线B"
正确："路线B最符合你的情况。但选择权在你。"

原则2：透明而非隐藏
错误："根据我们的算法，这是最佳选择"
正确："基于你的体力(6/10)、时间(5天)和偏好，
       路线B需要的体力(5.5/10)最匹配。"

原则3：人性化而非生硬
错误："数据显示风险评分为0.65"
正确："这条路线的风险是中等。这意味着你需要
       充足的准备，但完全在可控范围内。"

原则4：给予选择而非推荐
错误："这个路线最好"
正确："这条路线最好的地方是景观优美，
       但需要你的体力能达到7/10。"

原则5：尊重用户的判断
错误："你的选择可能有风险"
正确："你选择了高风险路线。这意味着...
       但我们已准备好应急方案。"
```

## 第八章：设计规范总结

### 8.1 核心设计原则检查清单

在任何设计决策中，问自己：

```
【决策支持性检查】
□ 这个设计帮用户理解信息了吗？
□ 这个设计支持用户的判断吗？
□ 还是这个设计在替用户做决策？

【信息透明性检查】
□ 用户能看到数据来自哪里吗？
□ 用户能理解数据的确定性吗？
□ 系统是否隐藏了任何限制或假设？

【尊重用户检查】
□ 用户有充分的选择吗？
□ 用户的选择被尊重了吗？
□ 系统是否强行推动用户？

【可访问性检查】
□ 不同背景的用户都能理解吗？
□ 不同能力的用户都能使用吗？
□ 是否有调整信息密度的选项？

【信任建立检查】
□ 界面看起来可靠吗？
□ 系统的行为是一致的吗？
□ 是否有任何让用户怀疑的地方？
```

| 采用分层信息架构 | 避免用户信息过载 | 需要多次点击才能获得完整信息 |
| 强调数据来源和置信度 | 建立信任，支持可解释性 | 界面会显得复杂 |
| 提供多个选项而非单一推荐 | 尊重用户判断，支持决策 | 用户可能更想要明确建议 |
| 突出风险而非隐藏风险 | 安全优先，帮助用户做知情决策 | 可能让用户感到不安 |
| 使用颜色编码风险等级 | 快速传达信息 | 色盲用户需要文字补充 |
| 提供撤销和修改选项 | 支持用户反思和改变 | 界面复杂度增加 |
| 支持多人格适配 | 满足不同场景的用户需求 | 系统复杂度增加 |
| 采用渐进式信息披露 | 平衡信息量和可用性 | 需要多次交互才能获得完整信息 |
