### 核心架构

**统一入口**：`POST /agent/route_and_run` → `agentApi.routeAndRun()`（定义在 `src/api/agent.ts`）

**路由类型**（`RouteType`）：
- **SYSTEM1_API**：快速 API 调用

**结果状态**（`ResultStatus`）：
- **OK**：成功
- **NEED_MORE_INFO**：需要更多信息
- **NEED_CONSENT**：需要用户同意
- **NEED_CONFIRMATION**：需要用户确认（审批流程）
- **FAILED**：失败
- **TIMEOUT**：超时

**UI 状态**（`UIStatus`）：
- `thinking`、`browsing`、`verifying`、`repairing`、`awaiting_consent`、`awaiting_confirmation`、`done`、`failed`

### 核心能力

**Should-Exist Gate（路线存在性门控）**：
- 执行位置：Agent 决策流程中（通过 `RouteAndRunResponse` 返回）
- 负责 Agent：GatekeeperAgent（Abu）
- 输出：通过 `DecisionLogItem` 记录决策过程
- 三人格评审：Abu（安全）、Dr.Dre（节奏）、Neptune（修复）
- **预算门控**：
  - 评估位置：规划工作台执行流程中（通过 `POST /planning-workbench/budget/evaluate` 接口）
  - 评估时机：方案生成后、提交前
  - 评估结果：`ALLOW`（允许）/ `NEED_ADJUST`（需要调整）/ `REJECT`（拒绝）
  - 决策日志：通过 `GET /planning-workbench/budget/decision-log` 获取
  - 优化建议：通过 `POST /planning-workbench/budget/apply-optimization` 应用
  - 参考接口：`src/api/planning-workbench.ts` - `evaluateBudget()`、`getBudgetDecisionLog()`、`applyBudgetOptimization()`

**可执行行程（Executable Itinerary）**：
- 交通班次/票务、POI、开放时间、预订链接、紧急点位

- 数据结构：`TripDetail`、`TripDay`、`ItineraryItem`（定义在 `src/types/trip.ts`）

[设计哲学：基于认知科学的人机交互原则]

你必须坚持以下原则（硬规则），这些原则基于**认知科学理论**和**人机交互研究**：

### 认知负荷理论（Cognitive Load Theory）

- **内在认知负荷**：通过信息架构和视觉层次降低用户理解信息的难度
  - 复杂决策信息应该分层展示（如决策日志默认折叠）
  - 澄清问题应该分组呈现，避免一次性展示过多问题
- **外在认知负荷**：通过清晰的界面设计和交互模式减少不必要的认知负担
  - 减少视觉噪音，突出关键信息
  - 使用一致的交互模式，降低学习成本
- **相关认知负荷**：通过渐进式披露和上下文帮助促进用户学习
  - 决策日志默认摘要，点击查看详情
  - 证据链默认折叠，需要时展开

### Miller's Law（米勒定律）

- 人类工作记忆容量有限（7±2 个信息块）
- **应用规则**：
  - 复杂信息应该分块呈现，每块不超过 7 个元素
  - 澄清问题应该分组展示，每组不超过 5 个问题
  - 快捷回复按钮应该限制数量（建议不超过 4-5 个）
  - 参考实现：`src/components/trips/NLChatInterface.tsx` 中的快捷回复按钮组

### Hick's Law（希克定律）

- 选择数量越多，决策时间越长
- **应用规则**：
  - 澄清问题的选项应该控制在合理范围内（建议不超过 5 个）
  - 快捷回复按钮应该限制数量，避免选择困难
  - 多步骤流程应该分解为多个简单决策

### Fitts's Law（费茨定律）

- 交互目标应该足够大，易于点击
- **应用规则**：
  - 关键操作按钮应该有足够的点击区域（建议最小 44×44px）
  - 常用操作应该放在易于到达的位置（屏幕底部、拇指热区）
  - 确认按钮应该足够突出和易于点击

### Progressive Disclosure（渐进式披露）

- 复杂信息不应该一次性展示，而是根据用户当前任务和上下文逐步呈现
- **应用规则**：
  - 决策日志默认折叠，需要时展开
  - 证据链默认摘要，点击查看详情
  - 预算评估结果默认显示摘要，详情在抽屉中展示
  - 这符合**认知科学**中的"分块处理"原理

### Affordance（示能性）

- 界面元素应该清晰表达其功能和使用方式
- **应用规则**：
  - 按钮应该看起来可点击（使用阴影、悬停效果）
  - 输入框应该看起来可输入（清晰的边框、占位符）
  - 状态应该通过视觉和交互反馈清晰表达（加载动画、状态图标）

### Feedback（反馈）

- 每个用户操作都应该有明确的反馈
- **应用规则**：
  - 反馈应该及时（< 100ms 的即时反馈，< 1s 的视觉反馈）
  - 反馈应该清晰（用户能理解发生了什么）
  - UI 状态转换应该有明确的视觉反馈（`thinking` → `browsing` → `done`）
  - 参考实现：`src/components/trips/NLChatInterface.tsx` 中的状态指示器

### Error Prevention & Recovery（错误预防与恢复）

- 通过设计减少用户犯错的可能性
- **应用规则**：
  - 实时验证输入（如预算约束设置）
  - 提供清晰的错误提示和恢复建议
  - 危险操作需要明确的确认步骤（如 `NEED_CONFIRMATION` 状态的审批流程）

### Accessibility（无障碍设计）

- 遵循 WCAG 2.1 AA 标准
- **应用规则**：
  - 确保键盘导航可用（Tab 键导航、Enter 键确认）
  - 提供足够的颜色对比度（文本对比度 ≥ 4.5:1）
  - 支持屏幕阅读器（ARIA 标签、语义化 HTML）
  - 交互元素应该有明确的焦点状态

### Consistency（一致性）

- 相同功能在不同页面应该有一致的交互模式
- **应用规则**：
  - 决策状态展示方式一致（三人格卡片、决策日志）
  - 确认流程一致（审批对话框、确认按钮位置）
  - 证据展示方式一致（证据抽屉、证据链结构）
  - 这符合**认知科学**中的"模式识别"原理

### 人机交互设计关注点

1. **认知负荷优化**
   - 信息分块呈现（Miller's Law）：澄清问题分组、决策日志折叠
   - 选择数量控制（Hick's Law）：快捷回复按钮限制、澄清问题选项限制
   - 渐进式披露机制：证据链默认摘要、预算评估结果默认折叠
   - 参考实现：`src/components/trips/NLChatInterface.tsx` 中的快捷回复和澄清问题组件

2. **交互模式设计**
   - 基于 Fitts's Law 的点击目标设计：按钮最小尺寸 44×44px
   - 基于 Affordance 的界面元素设计：清晰的视觉反馈、状态指示
   - 基于反馈理论的交互反馈设计：及时的状态更新、清晰的错误提示
   - 参考实现：`src/components/trips/NLChatInterface.tsx` 中的状态指示器和反馈机制

3. **可用性工程**
   - 可用性测试方法：认知走查、启发式评估、任务分析
   - 关键操作可用性：预算评估、决策确认、澄清问题回答
   - 错误预防与恢复：实时验证、确认机制、错误提示

4. **无障碍设计**
   - WCAG 2.1 AA 标准：颜色对比度、键盘导航、屏幕阅读器支持
   - 键盘导航支持：Tab 键导航、Enter 键确认、Esc 键取消
   - 屏幕阅读器支持：ARIA 标签、语义化 HTML、状态公告

5. **多模态交互**
   - 语音交互设计：语音输入、语音反馈
   - 手势交互设计：触控手势、滑动操作
   - 触控交互优化：触摸目标大小、手势识别
