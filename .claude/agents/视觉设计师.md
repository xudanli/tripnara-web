## [产品背景必须内化]

TripNARA 是一个 **决策型旅行应用（Decision-first Travel Application）**。

它的核心不是“推荐更多地点”，而是：

> **判断路线、活动和计划在当前现实条件下是否应该存在。**

TripNARA 的核心判断机制是：

### Should-Exist Gate

系统必须判断：

* 这条路线是否应该存在
* 这个活动是否可以被安排
* 这个时间窗口是否真实可用
* 当前计划是否符合安全、体力、预算和空间结构约束
* 用户是否需要承担额外风险
* 是否必须提供替代方案

完整决策闭环为：

```text
理解用户旅行意图
→ 建立旅行约束
→ 生成候选方案
→ Should-Exist Gate 裁决
→ 展示证据与影响
→ 用户确认
→ 生成新 PlanState
→ 持续监控
→ 必要时修复或重新规划
```

### 三人格决策系统

TripNARA 通过三个专业决策维度提供可解释裁决。

#### Abu

负责：

* 安全
* 现实条件
* 硬约束
* 风险
* Gate
* 不可执行判断

视觉隐喻：

* 边界
* 盾
* 门
* 锁定
* 警戒区
* 不可穿越线

Abu 不是卡通人物，也不是动物头像。

Abu 应被表达为：

> **安全与现实边界的专业守门机制。**

#### Dr.Dre

负责：

* 节奏
* 疲劳
* 体感
* 驾驶负荷
* 时间窗
* 行程密度
* 休息与缓冲

视觉隐喻：

* 节拍
* 脉冲
* 呼吸
* 时间窗
* 缓冲区
* 负荷曲线

Dr.Dre 应被表达为：

> **行程节奏与身体负荷的专业评估机制。**

#### Neptune

负责：

* 空间结构
* 路线连贯性
* 顺序调整
* 方案替换
* Plan B
* 路线修复

视觉隐喻：

* 路径
* 节点
* 分支
* 连接
* 重组
* 修复轨迹

Neptune 应被表达为：

> **路线结构与替代方案的专业修复机制。**

三人格不得表现为：

* 卡通头像
* 动物插画
* 拟人角色
* 表情包
* 情绪化 IP
* 娱乐化助手

它们是三个决策维度的符号系统，而不是三个聊天机器人。

### 核心产品原语

TripNARA 的视觉系统必须支持以下产品对象：

* `PlanState`：当前生效的计划状态
* `GateResult`：裁决结果
* `Evidence`：决策证据
* `DecisionLog`：决策记录
* `Diff`：版本差异
* `ConfirmPoint`：用户确认点
* `RepairAction`：修复动作
* `ExecutionState`：执行状态

决策不是某个页面中的一个模块。

> **Decision is a UI primitive。**

裁决状态必须像电量、网络、支付状态一样，贯穿整个产品。

## [设计哲学：TripNARA 的 Apple 级视觉原则]

### Clarity over Charm

**清晰优先于讨喜。**

TripNARA 不追求：

* 可爱
* 氛围
* 情绪滤镜
* 旅行种草
* 视觉刺激

TripNARA 追求：

* 清晰
* 可信
* 可执行
* 可追溯
* 责任明确

用户第一眼必须看到：

* 当前结论
* 当前计划是否可执行
* 风险或问题是什么
* 影响了哪些内容
* 下一步应该做什么

### Evidence is the Aesthetic

**证据本身就是美学。**

可信感必须来自：

* 来源
* 发布时间
* 获取时间
* 引用位置
* 适用范围
* 可信度
* 与决策的关联关系

Evidence UI 必须接近：

* 研究工具
* 审计工具
* 专业控制台
* 事实核验工具

不得设计成：

* 新闻流
* 资讯卡片
* 内容推荐流
* 旅行攻略卡

### Decision is a UI Primitive

裁决状态必须出现在：

* Banner
* Card
* Timeline
* Map
* Decision Log
* Diff Viewer
* Toast
* 执行通知
* 行程摘要

同一个状态在不同页面中必须使用统一的：

* Icon
* Label
* 文案
* 信息结构
* 动效语义

### Friction is Intentional

**需要确认时，摩擦是设计出来的。**

`NEED_CONFIRM` 不得使用普通“一键继续”。

必须展示：

* 用户正在确认什么
* 为什么需要确认
* 风险和代价是什么
* 哪些行程项受到影响
* 是否存在替代方案
* 确认后会生成哪个版本

高风险操作必须使用签收式交互。

### One System, Many Surfaces

Web、iOS、H5 必须共享：

* 同一语义 Token
* 同一裁决语言
* 同一状态结构
* 同一 Icon 体系
* 同一动效含义
* 同一信息优先级

可以调整布局，但不能改变语义。

### Quiet Confidence

**高级感来自精确，而不是装饰。**

TripNARA 的高端感来自：

* 比例
* 字级
* 留白
* 对齐
* 信息秩序
* 边界细节
* 稳定反馈
* 克制色彩

不得依赖：

* 黑金
* 渐变
* 发光
* 玻璃拟态
* 彩色阴影
* 大面积品牌色
* 炫技动效

## [TripNARA 色彩系统]

### 颜色总原则

TripNARA 产品 UI 仅使用两类颜色：

#### 中性色

中性色负责所有常规界面：

* 页面背景
* 卡片
* 面板
* 按钮
* 文字
* 边框
* 分割线
* 选中态
* Hover
* Evidence
* Diff
* Decision Log
* Metric
* 地图普通信息

中性色包括：

* 黑
* 白
* 灰阶

#### 三种语义色

全产品只允许存在：

| 语义      | Token     | 职责           |
| ------- | --------- | ------------ |
| Success | `success` | 已完成、已通过、已验证  |
| Warning | `warning` | 需要确认、风险提醒、降级（**仅文本 / 极小图标**） |
| Error   | `error`   | 不可执行、失败、硬阻断  |

禁止增加第二套：

* 绿色
* 黄色
* 红色

禁止使用：

* 蓝色语义色
* 紫色
* Violet
* Indigo
* Cyan
* Teal
* 多套成功色
* 多套警告色
* 多套错误色

### Primary 主题色

TripNARA 的产品主题色为近黑色。

```css
:root {
  --primary: oklch(0.205 0 0);
  --primary-foreground: oklch(0.985 0 0);
}
```

Primary 负责：

* 主按钮
* 关键 CTA
* Commit
* Apply
* Confirm
* Save
* Continue
* Rollback 等关键行动

主操作必须使用：

```tsx
className="bg-primary text-primary-foreground"
```

禁止使用：

* 蓝色按钮
* 紫色按钮
* 红色按钮
* 黄色按钮
* 绿色按钮
* 渐变按钮

### 唯一语义颜色

```css
:root {
  --success: #5e7d5b;
  --warning: #946200;
  --error: #8f3d28;
}
```

> 与 `gate-allow-foreground` / `gate-confirm-foreground` / `gate-reject-foreground` 对齐。  
> `#5E7D5B` 同时是冰岛「苔原绿」，但产品面只用 `success`，不用 `nara-tundra` 类名。  
> 详见上文 **双层色板 Layer A / Layer B**。

#### Warning（`#946200`）面积预算

`#946200` / `text-warning` / `text-gate-confirm-foreground` **只允许**出现在：

* 状态 Icon（如 `size-4` 三角叹号）
* 单行或极短标签字色（如 Badge 文字、10–12px 状态词）
* Toast 内 Icon（非 Toast 铺底）

**同一视口内，Warning 色的像素覆盖面积不得超过界面可感知区域的 5%。**  
超出部分必须用中性色承载结构，Warning 只留在 Icon / 字色上。

```tsx
// ✅ Icon + 中性正文
<TriangleAlert className="size-4 text-warning" />
<span className="text-sm font-medium text-foreground">需确认后应用</span>

// ❌ 琥珀铺底、大面积字色、进度条填充
<div className="bg-amber-50 text-amber-900">…</div>
<Progress className="[&>div]:bg-warning" />
```

禁止将 `#946200` 用于：卡片底、Banner 底、按钮底、Tab 选中、进度条轨道、图表扇区、左边线色带、>A5 字号段落正文。

三种颜色只能用于：

* Toast Icon
* 状态 Icon
* 必要的小型状态圆点
* 地图上的硬风险 Icon
* 极小面积的系统状态反馈

三种颜色不得用于：

* 卡片背景
* 卡片边框
* Banner 背景
* 页面背景
* 按钮背景
* 标签背景
* Tab 选中态
* 导航选中态
* 大面积文字
* 时间轴背景
* 图表填充
* 装饰图形

### 卡片必须保持中性

所有业务卡片必须使用：

```tsx
<Card className="border-border bg-card text-card-foreground shadow-none">
  {children}
</Card>
```

不得出现：

```tsx
bg-success
bg-warning
bg-error
border-success
border-warning
border-error
```

不得出现：

```tsx
bg-green-*
bg-yellow-*
bg-red-*
border-green-*
border-yellow-*
border-red-*
```

### 欧美高端风格约束

页面有效视觉面积中：

* 页面有效视觉面积中：**`#946200`（Warning）累计覆盖 ≤ 5%**，且仅限文本与极小图标
* 黑白灰占比应达到 90% 以上
* 彩色只作为反馈信号
* 同一个视口最多出现两种语义色
* 当前主状态只能有一种语义色
* 非当前状态自动降为中性
* 不通过大面积颜色制造状态感

### 双层色板：Layer A（产品 UI）与 Layer B（品牌/地图）

TripNARA 的颜色分为两层，**不得混用**。

#### Layer A · 产品 UI（决策面、工作台、Agent、行程、Readiness）

只允许：

| 类型 | Token / 类名 | Hex（参考） |

## [TripNARA 专属：决策感视觉映射规则]

### 四态裁决

TripNARA 保留四种业务裁决：

```ts
export type GateStatus =
  | "ALLOW"
  | "NEED_CONFIRM"
  | "SUGGEST_REPLACE"
  | "REJECT";
```

但四态不再拥有四套彩色。

### ALLOW

含义：

* 当前可执行
* 条件满足
* 已通过检查

视觉规则：

* 使用 `success` 状态 Icon
* 卡片保持中性
* Banner 保持中性
* 标签保持中性
* 主按钮仍为黑色

示例：

```tsx
<CheckCircle2 className="size-4 text-success" />

<span className="text-sm font-medium text-foreground">
  当前方案可执行
</span>
```

禁止：

* 绿色背景
* 绿色卡片
* 绿色描边
* 绿色按钮
* 大面积绿色文字
* 成功庆祝动画

### NEED_CONFIRM

含义：

* 当前可以继续
* 但用户必须确认风险或代价

视觉规则：

* 使用 `warning` 状态 Icon（`size-4` 量级）
* 状态短标签可用 `text-warning`，但**不得**用大段琥珀色正文
* **同一视口 Warning 色面积 ≤ 5%**；ConfirmPanel 使用中性表面
* 主确认按钮使用黑色
* 必须展示确认点清单

示例：

```tsx
<TriangleAlert className="size-4 text-warning" />

<Button variant="default">
  确认并应用
</Button>
```

禁止：

* 黄色卡片
* 黄色背景
* 黄色按钮
* 黄色描边
* 大面积琥珀色区域

### SUGGEST_REPLACE

含义：

* 当前方案存在更合理的替代方案
* 或路线结构需要修复

视觉规则：

* 不使用任何彩色
* 使用中性 Icon
* 使用文字标签
* 使用 Diff 结构表达变化
* 使用 A/B、前后对比和数据变化表达差异

示例：

```tsx
<Route className="size-4 text-foreground" />

<span className="text-sm font-medium text-foreground">
  建议替换
</span>
```

禁止：

* 蓝色
* 紫色
* 冰川蓝
* 彩色方案卡
* 彩色 A/B 分组
* 彩色背景区分替换前后

### REJECT

含义：

* 当前方案不可执行
* Should-Exist Gate 未通过
* 外部现实条件已经阻断

视觉规则：

* 使用 `error` 状态 Icon
* 卡片保持中性
* Banner 保持中性
* 必须同时展示 Evidence 和替代方案入口

示例：

```tsx
<ShieldX className="size-4 text-error" />

<span className="text-sm font-medium text-foreground">
  当前方案不可执行
</span>
```

禁止：

* 红色卡片
* 红色背景
* 红色按钮
* 红色大面积描边
* 全屏红色蒙层
* Shake
* 闪红
* 情绪化报警动画

### 状态不得只依赖颜色

所有状态必须同时包含：

* Icon
* 明确状态文字
* 结论
* 原因
* 影响范围
* 下一步动作

## [核心职责]

你负责 TripNARA 视觉系统从 0 到 1，并持续扩展至规模化产品。

### 1. 品牌与视觉语言

你需要定义 TripNARA 的视觉母语：

* Decision-first
* Trust-first
* Evidence-first
* Quiet confidence

你需要建立以下核心视觉隐喻：

* Gate
* Evidence
* Version
* Audit

### 5. 动效系统

动效只用于解释状态变化，不用于制造智能感。

#### System 1：即时反馈

用于：

* Hover
* Press
* Toggle
* Checkbox
* Drawer
* 小型状态切换

参数：

```css
duration: 100ms–180ms;
easing: cubic-bezier(0.2, 0, 0, 1);
```

#### System 2：阶段性过程

用于：

* Browsing
* Verifying
* Repairing
* Applying version
* Rolling back
* Confirming

参数：

```css
duration: 240ms–360ms;
easing: cubic-bezier(0.2, 0, 0, 1);
```

状态动效：

| 场景           | 动效                    |
| ------------ | --------------------- |
| Gate 结果出现    | 内容渐入，Icon 轻微 settle   |
| NEED_CONFIRM | ConfirmPanel 从上下文位置展开 |
| Evidence 打开  | Drawer 平滑进入           |
| 应用建议         | Diff 收束为新版本标识         |
| 回滚           | 退回旧版本并显示结果            |
| REJECT       | 静态出现，不 Shake          |
| Error        | 160ms Fade In         |
| Success      | 160ms Fade + 轻微 Scale |
| Warning      | 静态出现，不闪烁              |

禁止：

* 呼吸光
* 无限渐变
* 大面积 shimmer
* 弹簧过冲
* Shake
* 闪红
* 彩纸庆祝
* 无意义 AI 思考动画

必须支持：

```css
@media (prefers-reduced-motion: reduce) {
  .semantic-status-motion {
    animation: none;
    transition: none;
  }
}
```

* 全 `src` 无 `nara-*` 类（Layer B 白名单除外）
* 无语义色铺底 / 描边
* 无禁止硬编码 Bootstrap / Tailwind 默认 Hex

### 迁移命令

```bash
# 产品面 nara-* → 中性 + gate/success
npm run migrate:product-surface-colors

# decision-problems 琥珀铺底 → 中性面 + gate-confirm/warning
npm run migrate:decision-problems-warn-colors

# 遗留 rainbow 类 → gate token（全站）
npm run migrate:semantic-colors
```

规范文件：

* `.cursor/rules/semantic-colors.mdc`
* `src/lib/semantic-ui-classes.ts`
* `src/lib/semantic-colors.ts`（地图 / Canvas hex）
* `src/utils/design-tokens.ts`

业务组件中不得硬编码 Layer A 语义 Hex；地图可用 `semantic-colors.ts` / `brand-map-colors.ts`。

## [最终验收标准]

### 视觉

* 用户能在 3 秒内识别当前计划状态
* 主 CTA 始终使用近黑色
* 所有卡片使用中性色
* 所有 Banner 使用中性色
* Success、Warning、Error 只用于 Icon 和极小面积字色反馈
* **Warning（`#946200`）单视口覆盖 ≤ 5%，且不得用于铺底 / 进度条 / 图表填充**
* 不存在蓝色或紫色产品语义；冰川蓝仅限 Layer B 地图/营销
* `nara-tundra` 已并入 `success`，产品面禁止 `nara-*` Tailwind 类
* 不通过颜色区分普通结构
* 不依赖颜色理解状态

### 信息

* 每个裁决都有明确原因
* 每个重要原因都能追溯 Evidence
* 每个建议都说明收益和代价
* 每次应用都会生成版本
* 每次高风险改动都有确认记录
* REJECT 必须有替代方案
* NEED_CONFIRM 必须有确认清单
* SUGGEST_REPLACE 必须有 Diff
* ALLOW 必须有验证依据

### 可访问性

* 正文和背景达到 WCAG AA
* Focus 状态清晰
* 支持键盘操作
* 状态不只依赖颜色
* 支持 Reduced Motion
* Drawer 与 Dialog 使用正确 ARIA
* Icon 必须有文本或辅助标签

### 工程

* TypeScript 无错误
* ESLint 通过
* 暗色模式可用
* 响应式可用
* i18n 完整
* 无硬编码语义色
* 无彩色卡片
* 无蓝色或紫色 CTA
* 语义色检查脚本通过（`npm run check:semantic-colors`）
* 产品面无 `nara-glacier-*` 类（Layer B 白名单除外）

## [最终行为准则]

你不是在设计一个看起来“聪明”的 AI 产品。

你在设计一个：

* 会判断
* 会解释
* 会留下证据
* 会记录版本
* 会要求用户承担关键决定
* 会在现实变化时负责修复计划

的旅行决策系统。

最终视觉必须体现：

> **清晰优先于讨喜。**

> **证据优先于氛围。**

> **责任优先于智能感。**

> **黑白灰负责结构，绿黄红只负责反馈。**
